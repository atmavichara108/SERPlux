# SERPlux — Руководство пользователя

**Версия:** 1.0  
**Дата:** 2026-07-04  
**Статус:** Актуально

---

## Что такое SERPlux

SERPlux — система автоматического сбора поисковой выдачи (Google, Яндекс) через Topvisor API с разметкой тональности URL и выгрузкой в Google Sheets.

**Основная задача:** мониторинг позиций и репутации субъектов (персон, компаний) в поисковой выдаче по разным гео.

**Как работает:**
1. Вы запускаете сбор через меню в Google Sheets
2. Система собирает выдачу через Topvisor API
3. Размечает тональность каждого URL (позитивная/негативная/нейтральная)
4. Выгружает данные в плоский лист «Данные»
5. Строит матрицу-отчёт на листе «Отчёт»

**Интерфейс:** Google Sheets (меню SERPlux + лист «Настройки»). Веб-интерфейс не используется.

---

## Возможности

### Что умеет SERPlux

**Сбор данных:**
- Google, Яндекс (yandex.ru), Яндекс.com
- 6 гео: Литва, Германия, Великобритания, Лондон, ОАЭ, Кипр
- Глубина сбора: 10, 20, 50 или 100 позиций
- Идемпотентность: повторный прогон не дублирует данные

**Разметка тональности (двухрежимная система с версии 2026-07-10):**
- Режим `auto` (дефолт) — справочник доменов → LLM по сниппету → neutral при ошибке (иерархия)
- Режим `deep` — для v2, уточнение neutral результатов по контенту страницы (пока заглушка)
- Кэширование: уже размеченные пары (URL × query) берутся из справочника, LLM не вызывается
- Принудительная переразметка: опция `force_relabel` для повторного прохода через LLM
- Нейтральность как маркер: `neutral` — маркер неуверенности LLM, для разметки в режиме DEEP

**Выгрузка:**
- Плоский лист «Данные» с заголовками (Дата, ПС, Субъект, Гео, Позиция, URL, Домен, Сниппет, Метка)
- Новые данные вставляются сверху, старые уезжают вниз
- Идемпотентность: повторный экспорт за ту же дату пропускается

**Отчёт:**
- Матрица 16 колонок: позиции и URL для 4 субъектов + разделители
- Блоки по поисковым системам (Google, Яндекс, Яндекс.com)
- Секции по гео (Литва → Германия → Великобритания → Лондон → ОАЭ → Кипр)
- Цветовая индикация тональности: зелёный (positive), красный (negative), жёлтый (neutral)
- Глубина отчёта: 10 позиций (REPORT_DEPTH)

**Мультиклиентность:**
- Несколько клиентов в одной системе
- CRUD-профили клиентов через API
- Переключение через `client_id` на листе «Настройки»

**Версионность разметки:**
- Каждая разметка — новая версия (не перезаписывает старую)
- История меток сохраняется в БД
- Разные режимы разметки — независимые ветки версий

### Лист «Настройки»

Формат: таблица ключ–значение (колонка A = ключ, B = значение, C = подсказка).

| Ключ | Значение по умолчанию | Описание |
|------|----------------------|----------|
| `client_id` | (пусто) | ID клиента (латиница, напр. `sudheimer`) |
| `depth` | `10` | Глубина сбора: 10, 20, 50 или 100 |
| `with_labels` | `true` | Включить разметку: `true` или `false` |
| `label_mode` | `auto` | Режим разметки: `auto` (дефолт) или `deep` (v2) |
| `date` | `today` | Дата сбора: `today` или `YYYY-MM-DD` |
| `force_relabel` | `false` | Принудительная переразметка: `true` или `false` |
| `force_rebuild_report` | `false` | Перестроить отчёт: `true` или `false` |
| `report_date` | `latest` | Дата отчёта: `latest` или `YYYY-MM-DD` |
| `provider_chain` | `opencode-zen` | Цепочка провайдеров LLM (через запятую) |
| `status` | `idle` | Статус последнего прогона (обновляется автоматически) |

**Data Validation:** Для `depth` — список `10, 20, 50, 100`. Для `with_labels`, `force_relabel`, `force_rebuild_report` — список `true, false`. Для `label_mode` — список `auto, deep` (**старые значения domains/snippets/full больше не поддерживаются**). Для `client_id` — выпадающий список клиентов с сервера (обновляется через меню «Клиенты → Обновить список клиентов»). Для `provider_chain` — выпадающий список доступных провайдеров LLM (обновляется автоматически вместе со списком клиентов).

**Инициализация:** SERPlux → Настройки → [+] Инициализировать настройки — создаёт лист с шаблоном.

### Лист «Лог»

Журнал запусков. Создаётся автоматически при первом запуске.

**Колонки:**
- Дата/время
- Клиент
- Статус
- Сообщение
- Провайдер

Каждый запуск и проверка статуса добавляют строку в лог.

Плоский формат для фильтрации и анализа.

**Заголовки:**
```

```

**Пример:**
```

```

Новые данные вставляются сверху, старые уезжают вниз.

Матрица-pivot для визуального анализа.

**Структура:**
- 16 колонок (0-indexed)

- Блоки по поисковым системам (Google, Яндекс, Яндекс.com)

**Цветовая индикация:**
- Зелёный (#d4edda) — positive (позитивная тональность)
- Красный (#f8d7da) — negative (негативная тональность)
- Жёлтый (#fff3cd) — neutral (нейтральная тональность)

**Формат даты:** `D.M.YYYY` (например, `4.7.2026`).

## Как запустить прогон

### Первый запуск (настройка)

1. **Откройте таблицу** — меню SERPlux появится автоматически (через Installable Trigger).

2. **Инициализируйте настройки:**  
   SERPlux → Настройки → [+] Инициализировать настройки  
   Создаст лист «Настройки» с шаблоном ключей.

3. **Установите URL сервера:**  
   SERPlux → Настройки → Установить URL сервера  
   Введите URL: `https://<ваш-домен>`

4. **Установите секрет:**  
   SERPlux → Настройки → Установить секрет  
   Введите `WEBHOOK_SECRET` из `.env` на сервере.

5. **Загрузите список клиентов:**  
   SERPlux → Клиенты → Обновить список клиентов  
   Подтянет выпадающий список `client_id` с сервера.  
   Если список пуст — сначала добавьте клиента (см. раздел «Добавление клиента»).

6. **Выберите клиента:**  
   На листе «Настройки» выберите `client_id` из выпадающего списка.

### Запуск сбора

1. **SERPlux → ▶ Запустить сбор**

2. **Диалог подтверждения:**  
   Показывает параметры: клиент, глубина, разметка, дата, переразметка, перестроение отчёта, провайдер.  
   Нажмите «Да» для запуска.

3. **Прогон запущен:**  
   Диалог: «Прогон запущен. Начало: [время]. Проверьте статус через меню SERPlux → Проверить статус».

4. **Проверка статуса:**  
   SERPlux → ⟳ Проверить статус  
   Показывает: `running` (выполняется), `ok` (завершён), `error` (ошибка).

5. **Результат:**  
   - Лист «Данные» — плоский формат с заголовками
   - Лист «Отчёт» — матрица с цветовой индикацией
   - Лист «Лог» — запись о запуске

### Сбор за конкретную дату

1. На листе «Настройки» установите `date` в формате `YYYY-MM-DD`.
2. SERPlux → ▶ Запустить сбор.
3. Система запросит снимок Topvisor за указанную дату (если доступен) и соберёт выдачу.

**Примечание:** Topvisor хранит снимки ограниченное время. Если снимок за дату отсутствует — сбор вернёт пустой результат. Для уже собранных данных используйте разметку/отчёт без повторного сбора.

### Разметка без повторного сбора (label_only)

Если данные уже собраны, но нужно разметить их (например, другим режимом или с force_relabel):

1. На листе «Настройки» выберите режим `label_mode` и `force_relabel`.
2. **SERPlux → Разметить собранные данные**
3. Подтвердите запуск.

Сервер не будет собирать данные заново, а разметит уже сохранённые позиции для текущего клиента.

### Разметка за конкретную дату

1. **SERPlux → Разметить за дату...**
2. Выберите дату из списка доступных (GET /clients/{id}/dates).
3. Выберите режим разметки (`domains`, `snippets`, `full`).
4. Подтвердите принудительную переразметку (да/нет).
5. Подтвердите запуск.

### Построение отчёта за дату

Если данные уже собраны и нужно только построить отчёт:

1. **SERPlux → [>] Построить отчёт за дату...**

2. **Диалог выбора даты:**  
   Показывает список доступных дат для клиента. Введите дату (`YYYY-MM-DD`) или оставьте пустым для последней доступной.

3. **Подтверждение:**  
   Нажмите «Да».

4. **Результат:**  
   Отчёт построен на листе «Отчёт» за указанную дату.

### Добавление клиента

1. **SERPlux → Клиенты → Добавить клиента...**

2. **Шаги:**
   - Введите `client_id` (латиница, без пробелов).
   - Введите `client_name` (отображаемое имя).
   - Введите `project_id` в Topvisor.
   - Введите `sheet_id` Google Таблицы (опционально).
   - Выберите гео из списка регионов Topvisor (нумерованный список, можно выбрать несколько через запятую). Если Topvisor недоступен (502) — появится fallback на ручной ввод.
   - Выберите поисковики (`google`, `yandex_ru`, `yandex_com`).

3. **Результат:**  
   Клиент создан на сервере с указанными `geos` и `searchers`.  
   Не забудьте загрузить карту регионов на сервер: `regions_map_<client_id>.json`.

4. **Обновите список клиентов:**  
   SERPlux → Клиенты → Обновить список клиентов — чтобы новый клиент появился в выпадающем списке.

### Обновление гео клиента из Topvisor

Если у существующего клиента изменился набор регионов в Topvisor:

1. На листе «Настройки» выберите `client_id`.
2. **SERPlux → Клиенты → Обновить гео из Topvisor...**
3. Выберите гео из списка.
4. Подтвердите — сервер обновит профиль клиента (PUT /clients/{id}).

## Параметры прогона

### depth (глубина сбора)

**Значения:** `10`, `20`, `50`, `100`  
**По умолчанию:** `10`

Определяет, сколько позиций собирать из выдачи.  
- `10` — топ-10 (быстро, достаточно для большинства задач)
- `50` — топ-50 (Яндекс собирает до 50, Google до 10)
- `100` — топ-100 (для глубокого анализа)

**Влияние:** больше глубина = больше данных = дольше сбор = больше токенов на разметку.

### with_labels (включить разметку)

**Значения:** `true`, `false`  
**По умолчанию:** `true`

Если `false` — разметка тональности пропускается. Данные сохраняются без меток.  
Используется для быстрого прогона без LLM (нулевая стоимость).

### label_mode (режим разметки)

**Значения:** `auto`, `deep`  
**По умолчанию:** `auto`

**Внимание:** C версии от 2026-07-10 система перейдена на двухрежимную модель (вместо domains/snippets/full).  
См. раздел «Режимы разметки тональности».

### force_relabel (принудительная переразметка)

**Значения:** `true`, `false`  
**По умолчанию:** `false`

Если `true` — кэш меток игнорируется, все строки размечаются заново.  
Используется при исправлении ошибок разметки или смене провайдера.

**Влияние:** больше токенов (все URL переразмечаются через LLM).

### date (дата сбора)

**Значения:** `today` или `YYYY-MM-DD`  
**По умолчанию:** `today`

Дата сбора выдачи или разметки уже собранных данных.  
- `today` — сегодняшняя дата
- `YYYY-MM-DD` — конкретная дата (например, `2026-07-01`)

**Где используется:**
- При полном сборе (`▶ Запустить сбор`) — запрашивает снимок Topvisor за указанную дату.
- При разметке (`Разметить за дату...`) — размечает данные только за указанную дату.

**Примечание:** Topvisor хранит снимки ограниченное время. Если снимок за прошлую дату отсутствует — сбор вернёт пустой результат.

### force_rebuild_report (перестроить отчёт)

**Значения:** `true`, `false`  
**По умолчанию:** `false`

Если `true` — отчёт за указанную дату перестраивается с нуля, даже если он уже существует.  
Используется при исправлении данных или изменении логики отчёта.

### provider_chain (цепочка провайдеров LLM)

**Значения:** `zen` или другой ID провайдера из `GET /providers`; можно указать несколько через запятую  
**По умолчанию:** пусто (используется провайдер по умолчанию)

Определяет, какой провайдер(ы) LLM использовать для разметки.  
- Выпадающий список обновляется через `Клиенты → Обновить список клиентов`.
- Если указать несколько провайдеров через запятую — используется фолбек-цепочка (при ошибке первого пробуется следующий).

### report_date (дата отчёта)

**Значения:** `latest` или `YYYY-MM-DD`  
**По умолчанию:** `latest`

Дата для построения отчёта (при `report_only=true`).  
- `latest` — последняя доступная дата из БД
- `YYYY-MM-DD` — конкретная дата

## Режимы разметки тональности

SERPlux размечает каждый URL по тональности: `positive` (позитивная), `negative` (негативная), `neutral` (нейтральная).

### Режим `auto` (иерархический, дефолт)

**Стоимость:** нулевая до LLM, если домен известен; ~0.4 цента/запрос за LLM только для новых  
**Скорость:** мгновенная для кэша; ~1 сек/запрос для новых + пауза 1с между вызовами

Режим **AUTO** выполняет иерархическую разметку с fallback на нейтральность:

1. **Шаг 1: Справочник доменов (нулевая стоимость)**  
   Для каждого URL берётся домен и сверяется со справочником `domain_labels` в БД.
   - Если домен есть в справочнике → используется его `sentiment` (`confidence='high'`)
   - LLM не вызывается, нулевая стоимость
   - Ручная разметка L1 (source=`manual_l1`) никогда не перезаписывается автоматикой

2. **Шаг 2: LLM по сниппету (если домен неизвестен)**  
   LLM получает заголовок + описание страницы из сниппета Topvisor.
   - Успех → `sentiment` из LLM, сохраняется в справочник с `source='snippet'`
   - `confidence='high'`
   - Промпт: «Оцени, как эта ссылка влияет на репутацию субъекта. Ответь ОДНИМ словом: positive/negative/neutral»

3. **Шаг 3: Fallback на neutral при ошибке (маркер неуверенности)**  
   При ошибке LLM (сеть, таймаут, провайдер недоступен):
   - `sentiment='neutral'`, `confidence='uncertain'`
   - neutral как явный маркер неуверенности системы (не окончательный результат)
   - Пустой сниппет → пропускаем с WARNING в логе

**Кэширование:** уже размеченные пары (URL + query) берутся из справочника повторно, LLM не вызывается.  
**force_relabel:** если `true` — кэш игнорируется, все строки переразмечаются через LLM.

**Когда использовать:** все случаи (дефолт). Масса → в процентах по доменам, экономия на LLM.

### Режим `deep` (зарезервирован для v2, заглушка)

**Статус:** заглушка (не реализован), подготовка к v2  
**Стоимость:** значительно выше (HTTP-запрос к каждому URL + полный контекст для LLM)

Режим **DEEP** обрабатывает только строки, где `sentiment='neutral'` (маркеры неуверенности).

1. **Фильтр:** берёт только позиции с `sentiment='neutral'`
2. **V2 обработка (в разработке):** HTTP-запрос к каждому URL, LLM по полному тексту страницы
3. **Сохранение:** результат с `source='page'`, `confidence='high'` (переписывает только neutral из AUTO)

**Сейчас:** режим возвращает строки без изменений (заглушка).

**Когда использовать (в v2):** после AUTO для уточнения неуверенных (neutral) результатов. Высокая стоимость, используется выборочно.

---

### Старые режимы (deprecated, из истории)

До версии от 2026-07-10 использовалась трёхрежимная система: `domains`, `snippets`, `full`.

**Миграция на новую систему:**
- `domains` → **`auto`** (шаг 1: справочник, нулевая стоимость)
- `snippets` → **`auto`** (шаги 2–3: LLM + fallback)
- `full` → **`deep`** (для v2, заход на страницу)

Старые значения `label_mode` в листе «Настройки» **не поддерживаются** (валидация отклонит). Обновите на `auto` или `deep`.

## Клиенты

SERPlux поддерживает несколько клиентов в одной системе.

### Профиль клиента

Каждый клиент описывается записью в таблице `clients`:
- `client_id` — уникальный slug (латиница, напр. `sudheimer`)
- `client_name` — отображаемое имя (напр. `Sudheimer Group`)
- `project_id` — ID проекта в Topvisor
- `sheet_id` — Google Sheet ID таблицы клиента (опционально)
- `searchers` — список поисковых систем: `google`, `yandex_ru`, `yandex_com`
- `geos` — список гео из карты регионов (напр. `Литва`, `Германия`)
- `regions_map` — имя файла карты регионов (опционально)

### Показать список клиентов

**SERPlux → Клиенты → Показать список клиентов**

Показывает всех зарегистрированных клиентов: ID, имя, project_id, sheet_id.

### Обновить список клиентов

**SERPlux → Клиенты → Обновить список клиентов**

GET /clients → обновляет выпадающий список `client_id` на листе «Настройки» и выпадающий список `provider_chain`.

### Добавить клиента

**SERPlux → Клиенты → Добавить клиента...**

Многошаговый диалог:
1. Введите ID клиента (латиница, без пробелов)
2. Введите отображаемое имя клиента
3. Введите project_id в Topvisor
4. Введите Google Sheet ID (опционально)
5. Выберите гео из списка регионов Topvisor (или вручную, если Topvisor недоступен)
6. Выберите поисковики (`google`, `yandex_ru`, `yandex_com`)

После добавления: «Не забудьте загрузить карту регионов на сервер: `regions_map_<client_id>.json`».

### Обновить гео из Topvisor

**SERPlux → Клиенты → Обновить гео из Topvisor...**

Для существующего клиента: GET /topvisor/regions → выбор гео → PUT /clients/{id}.

**SERPlux → Настройки → Показать текущий профиль**

Показывает профиль клиента из листа «Настройки» (поле `client_id`): ID, имя, project_id, sheet_id, даты создания/обновления.

### Переключение между клиентами

На листе «Настройки» измените `client_id` на нужный. При запуске сбора сервер подтянет профиль клиента из БД.

**Рекомендация:** один клиент = одна Google Таблица (проще, изоляция данных).

## История и версионность

### Версионность разметки

Каждая разметка — новая версия метки (не перезаписывает старую).

**Пример:**
- Прогон 1: режим `domains`, версия 1 → `sentiment=positive`
- Прогон 2: режим `snippets`, версия 1 → `sentiment=negative`
- Прогон 3: режим `snippets`, версия 2 (force_relabel) → `sentiment=neutral`

Все три версии сохраняются в БД. Разные режимы разметки — независимые ветки версий.

**Как получить последнюю метку:** `get_history()` возвращает последнюю метку на позицию (JOIN + MAX(created_at)).

**Как получить все версии:** фильтр `label_version='all'` в `get_history()`.

### История прогонов

Лист «Лог» содержит историю всех запусков:
- Дата/время
- Клиент
- Статус (`running`, `ok`, `error`)
- Сообщение
- Провайдер LLM

Лист «Данные» содержит плоский лог всех собранных данных (новые сверху, старые снизу).

## Безопасность

### Секреты

Все секреты хранятся в `.env` на сервере (не в коде):
- `WEBHOOK_SECRET` — токен авторизации API
- `OPENCODE_API_KEY` — ключ LLM-провайдера
- `TOPVISOR_USERNAME`, `TOPVISOR_PASSWORD`, `TOPVISOR_PROJECT_ID` — доступ к Topvisor
- `GOOGLE_SHEET_ID`, `GOOGLE_CREDENTIALS_PATH` — доступ к Google Sheets

**Важно:** `.env` в `.gitignore`, никогда не коммитится в репозиторий.

### Авторизация API

Все API-эндпоинты (кроме `/health`) требуют Bearer-токен:
```
Authorization: Bearer <WEBHOOK_SECRET>
```

Без токена — `401 Unauthorized`. С неверным токеном — `403 Forbidden`.

### Script Properties

`WEBHOOK_URL` и `WEBHOOK_SECRET` хранятся в Script Properties Google Sheets:
- **SERPlux → Настройки → Установить URL сервера**
- **SERPlux → Настройки → Установить секрет**

Script Properties **общие** для всех пользователей таблицы (установи один раз).

### API-ключи провайдеров

API-ключи LLM-провайдеров **не передаются** через UI. Они задаются в `.env` на сервере. В профиле провайдера хранится только имя переменной (`api_key_env_var`).

## Часто задаваемые вопросы

### Прогон уже выполняется, что делать?

Если запустить сбор пока идёт другой прогон — получите `409 Conflict`: «Прогон уже выполняется, подождите завершения».  
Проверьте статус: SERPlux → ⟳ Проверить статус. Дождитесь `ok` или `error`.

### Ошибка «Секрет не задан»

Запустите: SERPlux → Настройки → Установить секрет.  
Введите `WEBHOOK_SECRET` из `.env` на сервере.

### Ошибка «URL сервера не задан»

Запустите: SERPlux → Настройки → Установить URL сервера.  
Введите URL: `https://<ваш-домен>`.

### Ошибка «Выберите клиента»

На листе «Настройки» заполните поле `client_id` (например, `sudheimer`).

### Разметка не работает (все метки None)

**Режим `domains`:** справочник `domain_labels` пуст. Заполните справочник через API или вручную в БД.  
**Режим `snippets`:** проверьте `OPENCODE_API_KEY` в `.env` на сервере. Проверьте логи: `docker compose logs --tail 50`.

### Отчёт не строится

Проверьте, что данные есть в БД: SERPlux → ⟳ Проверить статус (должен быть `ok`).  
Проверьте логи: `docker compose logs --tail 50`.

### Как переразметить данные?

На листе «Настройки» установите `force_relabel=true`. Запустите сбор.  
Все строки будут переразмечены (кэш игнорируется).

### Как построить отчёт за прошлую дату?

SERPlux → [>] Построить отчёт за дату...  
Введите дату (`YYYY-MM-DD`) или оставьте пустым для последней доступной.
