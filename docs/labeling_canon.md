# Канон разметки SERPlux

Единый источник истины по тональности, источникам меток и режимам разметки.
Все модули (labeler, storage, webhook, exporter, Apps Script) должны
соответствовать этому документу. Расхождения заносятся в `docs/techdebt.md`.

## 1. Источник истины

Эталон Михаила, импортированный из листа «Эталон разметки» в таблицу
`domain_labels` с `source = 'manual_l1'` (~420 записей).

- Импорт выполняется функцией `importEtalonToDb()` в `apps_script.gs`.
- На сервере принимается батчами через `POST /labels/import` (`webhook.py`).
- Каждая запись идемпотентна по PRIMARY KEY `(domain, query, geo)`.

## 2. Допустимые значения тональности (sentiment)

Ровно три значения, совпадающих с CHECK-ограничением в БД:

- `positive` — позитивная тональность.
- `negative` — негативная тональность.
- `neutral` — неуверенность / невозможность однозначно определить тональность.
  **neutral — легитимное значение, не ошибка.**

Любые другие значения (например, `mixed`, `unknown`) отклоняются на этапе
импорта и не должны появляться в `domain_labels` / `labels`.

## 3. Источники меток (source) и приоритет

В `domain_labels` допустимы три источника:

1. `manual_l1` — эталон Михаила (высший приоритет).
2. `snippet` — автоматическая разметка по сниппету (режим `auto`).
3. `page` — автоматическая разметка по содержимому страницы (режим `deep`).

### Правила приоритета

- `manual_l1` **не перезаписывается** источниками `snippet` и `page`.
- `snippet` и `page` могут перезаписывать друг друга.
- `manual_l1` может перезаписать любую существующую запись.

Проверено на живой БД: записи с `source = 'manual_l1'` защищены от
перезаписи автоматикой. Логика реализована в `storage.upsert_domain_label()`
и `storage.bulk_upsert_domain_labels()`.

## 4. Режимы разметки (label_mode)

Для новых прогонов допустимы **ровно два** режима:

- `auto` — разметка по кэшу доменов + сниппету (быстро, дешево).
- `deep` — разметка по содержимому страницы (медленнее, дороже).

Режимы `domains`, `snippets`, `full` — устаревшая логика, **запрещены**
для новых операций. Они остаются в схеме БД и в `storage.LABEL_MODES`
только для обратной совместимости со старыми метками (см. `docs/techdebt.md`).

### Где режим задаётся и валидируется

- `webhook.py`: `RunRequest.label_mode` валидируется на `{auto, deep}`.
- `main.py`: дефолт `label_mode = config.get("label_mode", "auto")`.
- `labeler.py`: `label_mode` строго `auto` или `deep`.
- `apps_script.gs`: dropdown в листе «Настройки» содержит только `["auto", "deep"]`.

## 5. Связь слоёв

```
Сбор (collector) → positions (сырые данные: URL, сниппет, позиция)
                        ↓
Разметка (labeler) → labels (версионированные метки: sentiment,
                              label_mode ∈ {auto, deep})
                        ↓
Справочник (domain_labels) → источник для режима auto:
                              manual_l1 > snippet > page
                        ↓
Экспорт (exporter) → лист «Данные» (кэш выдачи для визуальной проверки)
Отчёт (reporter)   → лист «Отчёт» (только отчёт, без кэша)
```

## 6. Импорт новых эталонных записей

Формат батча для `POST /labels/import`:

```json
[
  {"domain": "example.com", "query": "juri sudheimer", "geo": "Литва", "sentiment": "positive"}
]
```

- `source` по умолчанию `manual_l1`.
- Битые записи пропускаются, остальной батч продолжается.
- Приоритет `manual_l1` сохраняется.

## 7. Что делать при расхождениях

1. Проверить CHECK-ограничения и множества констант в `storage.py`.
2. Если расхождение безопасно исправить (например, дефолтный `label_mode`) —
   исправить.
3. Если исправление ломает обратную совместимость (например, удаление
   устаревших режимов из CHECK `labels.label_mode`) — занести в
   `docs/techdebt.md` с планом миграции.
