## 1. Инвентаризация параметров

Каждый параметр, который должен быть кастомизируемым в системе.

### 1.1 Идентификация клиента/проекта

| Параметр | Тип | Допустимые значения | По умолчанию | Обязателен | Частота изменения |
|---|---|---|---|---|---|
| `client_id` | string | slug клиента, напр. `sudheimer`, `client2` | — | да | при смене клиента |
| `project_id` | int | ID проекта в Topvisor | — | да | при смене клиента |
| `client_name` | string | Отображаемое имя клиента | — | да | редко |

**Примечание:** `project_id` — внутренний идентификатор проекта в Topvisor. Сейчас захардкожен в `topvisor.py`. При мультиклиентности должен браться из профиля клиента.

---

### 1.2 Параметры сбора

| Параметр | Тип | Допустимые значения | По умолчанию | Обязателен | Частота изменения |
|---|---|---|---|---|---|
| `regions_map` | string (имя файла) | `regions_map.json`, `regions_map_client1.json`, … | `regions_map.json` | да | при смене клиента |
| `searchers` | list[enum] | `google`, `yandex_ru`, `yandex_com` | все три | нет | при смене клиента |
| `geos` | list[string] | ключи из regions_map (напр. `Литва`, `Германия`) | все из regions_map | нет | при смене клиента |
| `depth` | enum(int) | `10`, `20`, `50`, `100` | `10` | нет | каждый прогон |
| `date` | string (ISO) или `today` | `today` или `YYYY-MM-DD` | `today` | нет | каждый прогон |
| `timeout_sec` | int | 60–3600 | `900` | нет | редко |

**Примечание по `date`:** Параметр `date` позволяет запросить выдачу за конкретную прошлую дату. **Требует проверки:** поддерживает ли Topvisor Snapshots API получение снимка за произвольную прошлую дату, или только за сегодня/последний доступный снимок. Если не поддерживает — параметр остаётся только для фильтрации при построении отчёта из уже собранных данных в БД.

---

### 1.3 Параметры разметки тональности

| Параметр | Тип | Допустимые значения | По умолчанию | Обязателен | Частота изменения |
|---|---|---|---|---|---|
| `with_labels` | bool | `true`, `false` | `true` | нет | каждый прогон |
| `label_mode` | enum | `domains`, `snippets`, `full` | `snippets` | нет | редко |
| `force_relabel` | bool | `true`, `false` | `false` | нет | каждый прогон |

**Описание режимов `label_mode` (трёхэтапная модель):**

- **`domains`** — Режим 1: разметка по спискам доменов. Быстро, без LLM. Домен сверяется с заранее составленными списками (позитивные/негативные/нейтральные домены клиента). Нет совпадения → `neutral`. Стоимость: нулевая.
- **`snippets`** — Режим 2 (текущий): разметка по сниппету из выдачи. LLM получает заголовок + описание страницы из снимка Topvisor. Стоимость: ~0.4 цента/запрос (Zen qwen3.6-plus). **Текущий дефолт.**
- **`full`** — Режим 3: заход на страницу. LLM получает полный текст страницы (требует HTTP-запроса к каждому URL). Медленно, дорого, риск блокировок. Стоимость: значительно выше.

**Примечание по `force_relabel`:** При `false` (дефолт) labeler использует кэш — уже размеченные URL×query не перезапрашиваются. При `true` — кэш игнорируется, все строки размечаются заново.

### 1.4 Параметры отчёта

| Параметр | Тип | Допустимые значения | По умолчанию | Обязателен | Частота изменения |
|---|---|---|---|---|---|
| `report_depth` | enum(int) | `10`, `20`, `50` | `10` | нет | редко |
| `force_rebuild_report` | bool | `true`, `false` | `false` | нет | каждый прогон |
| `report_date` | string (ISO) или `latest` | `latest` или `YYYY-MM-DD` | `latest` | нет | каждый прогон |

**Примечание по `force_rebuild_report`:** Сейчас reporter.py пропускает построение, если отчёт за дату уже есть на листе (идемпотентность). `force_rebuild_report=true` принудительно перестраивает отчёт за указанную дату.

### 1.5 Параметры профиля клиента (раскладка колонок)

| Параметр | Тип | Допустимые значения | По умолчанию | Обязателен | Частота изменения |
|---|---|---|---|---|---|
| `subject_blocks` | list[object] | список субъектов с координатами колонок | — | да | редко |
| `cols_total` | int | кол-во колонок в матрице | `16` | да | редко |
| `geo_order` | list[string] | порядок гео в отчёте | из regions_map | нет | редко |
| `geo_display` | dict | маппинг ключей гео → отображаемые названия | — | нет | редко |
| `sheet_data_name` | string | имя листа для плоских данных | `Данные` | нет | редко |
| `sheet_report_name` | string | имя листа для матрицы-отчёта | `Отчёт` | нет | редко |
| `sheet_settings_name` | string | имя листа настроек | `Настройки` | нет | редко |

### 1.6 Параметры LLM-провайдеров

Каждый провайдер LLM описывается профилем с этими параметрами:

| Параметр | Тип | Допустимые значения | По умолчанию | Обязателен | Частота изменения |
|---|---|---|---|---|---|
| `provider_id` | string | slug: `zen`, `openai`, `gemini`… | — | да | редко |
| `provider_name` | string | отображаемое имя | — | да | редко |
| `endpoint` | string | URL API провайдера (OpenAI-совместимый) | — | да | редко |
| `model` | string | название модели | — | да | при смене модели |
| `api_key_env_var` | string | имя переменной в `.env` | — | да | редко |
| `enabled` | bool | `true`, `false` | `true` | нет | каждый прогон |
| `priority` | int | 1, 2, 3… (порядок в фолбек-цепочке) | 1 | нет | редко |
| `cost_per_1k` | float | стоимость за 1K входных токенов (USD) | 0 | нет | редко |

**Принцип фолбек-цепочки:** Провайдеры упорядочены по `priority`. При вызове LLM используется провайдер с priority=1. Если он недоступен (ошибка сети, 429, таймаут) — автоматически пробуется следующий по приоритету. Цепочка настраивается на уровне клиента (`provider_chain` в профиле клиента).

**Мониторинг:** Для каждого провайдера ведётся статистика: кол-во запросов, успешных/ошибочных, общая стоимость, средняя задержка. Доступна через API `GET /providers/{id}/stats`.

## 2. Раскладка: Sheets-меню vs API vs конфиг

### Принцип разделения

- **Конфиг/профиль клиента** — параметры, которые меняются редко (при смене клиента или реструктуризации). Хранятся в БД или JSON-файлах. Заказчик не трогает их при каждом прогоне.
- **Sheets-меню** — параметры, которые заказчик меняет вручную перед запуском. Читаются из листа «Настройки» или вводятся в диалоге.
- **API-тело** — параметры, которые передаются программно при вызове `/run`. Дублируют Sheets-параметры для автоматизации.

### Таблица распределения

| Параметр | Слой | Обоснование |
|---|---|---|
| `client_id` | Sheets-меню + API | Заказчик выбирает клиента перед каждым прогоном; API должен поддерживать автоматизацию по клиенту |
| `project_id` | Конфиг клиента | Технический ID Topvisor — не меняется, хранится в профиле |
| `regions_map` | Конфиг клиента | Привязан к клиенту; при выборе клиента подставляется автоматически |
| `searchers` | Конфиг клиента | Набор ПС определяется при настройке клиента, не при каждом прогоне |
| `geos` | Конфиг клиента | Аналогично searchers |
| `depth` | Sheets-меню + API | Заказчик может менять глубину для разных прогонов (быстрый топ-10 vs полный топ-50) |
| `date` | Sheets-меню + API | Нужен для ретроспективных прогонов; заказчик указывает вручную |
| `timeout_sec` | Конфиг клиента | Технический параметр, меняется редко |
| `with_labels` | Sheets-меню + API | Заказчик может отключить разметку для быстрого прогона |
| `label_mode` | Sheets-меню + API | Выбор режима разметки — осознанное решение заказчика |
| `force_relabel` | Sheets-меню + API | Нужен при исправлении ошибок разметки |
| `report_depth` | Конфиг клиента | Определяется форматом отчёта клиента, меняется редко |
| `force_rebuild_report` | Sheets-меню + API | Нужен при ручном исправлении отчёта |
| `report_date` | Sheets-меню + API | Для построения отчёта за конкретную дату из истории |
| `subject_blocks` | Конфиг клиента | Раскладка колонок — структурный параметр клиента |
| `cols_total` | Конфиг клиента | Аналогично subject_blocks |
| `geo_order` | Конфиг клиента | Порядок гео в отчёте — структурный параметр |
| `geo_display` | Конфиг клиента | Маппинг названий — структурный параметр |
| `sheet_*_name` | Конфиг клиента | Имена листов — настраиваются один раз |
| `provider_id` | Конфиг + API | Выбор провайдера для прогона; дефолт из профиля клиента |
| `provider_name` | Конфиг | Отображаемое имя, не меняется при прогоне |
| `endpoint` | Конфиг | Технический URL API, меняется редко |
| `model` | Конфиг + API | Можно менять на лету для A/B тестов |
| `api_key_env_var` | Конфиг | Привязка к `.env`, настраивается один раз |
| `enabled` | Конфиг + API | Вкл/выкл провайдера для конкретного прогона |
| `priority` | Конфиг | Порядок в фолбек-цепочке, определяется админом |
| `cost_per_1k` | Конфиг | Справочная стоимость, используется для мониторинга |

## 3. Модель мультиклиентности

### 3.1 Текущее состояние

Сейчас система однопользовательская: параметры клиента захардкожены в `config.py` (`SUBJECT_BLOCKS`, `GEO_ORDER`, `GEO_DISPLAY`) и в `main.py` (`DEFAULT_CONFIG`). Карта регионов — отдельный JSON-файл (`regions_map.json`). Для второго клиента уже есть `regions_map_client2.json`, но переключение требует ручной правки кода или переменной окружения `REGIONS_MAP`.

### 3.2 Целевая модель: профили клиентов

**Профиль клиента** — набор всех параметров, специфичных для одного клиента. Хранится как запись в таблице `clients` в SQLite (или как JSON-файл `profiles/<client_id>.json`).

#### Структура профиля

```
ClientProfile:
  client_id:        string          # уникальный slug, напр. "sudheimer"
  client_name:      string          # отображаемое имя, напр. "Sudheimer Group"
  project_id:       int             # ID проекта в Topvisor
  regions_map_file: string          # имя файла карты регионов
  searchers:        list[string]    # ["google", "yandex_ru", "yandex_com"]
  geos:             list[string]    # ключи гео из regions_map
  geo_order:        list[string]    # порядок гео в отчёте
  geo_display:      dict            # маппинг ключей → отображаемые названия
  subject_blocks:   list[object]    # субъекты с координатами колонок
  cols_total:       int             # ширина матрицы
  report_depth:     int             # глубина матрицы (10/20/50)
  sheet_id:         string          # Google Sheet ID клиента
  sheet_data_name:  string          # имя листа данных
  sheet_report_name: string         # имя листа отчёта
  timeout_sec:      int             # таймаут сбора
  provider_chain:     list[string]    # фолбек-цепочка провайдеров по приоритету, напр. ["zen", "openai"]
  created_at:       string (ISO)
  updated_at:       string (ISO)
```

### 3.3 Переключение между клиентами

**В Google Sheets:**

1. На листе «Настройки» есть строка `client_id` с выпадающим списком (Data Validation) из всех зарегистрированных клиентов.
2. При выборе клиента Apps Script подгружает его профиль (через GET `/clients` или из кэша в Script Properties) и показывает текущие параметры в информационной панели.
3. Запуск прогона (`▶ Запустить сбор`) передаёт `client_id` в теле запроса — сервер сам подтягивает нужный профиль.

**Альтернатива (проще):** Каждый клиент — отдельная Google Таблица со своим Apps Script. Переключение = открыть другую таблицу. Профиль клиента хранится в листе «Настройки» этой таблицы. Минус: дублирование кода Apps Script.

**Рекомендация:** Начать с «одна таблица на клиента» (проще, нет зависимостей). Объединить в одну таблицу с выпадающим списком клиентов — на следующем этапе, когда клиентов станет 3+.

### 3.4 Добавление нового клиента

Процесс без правки `.env` и пересборки Docker:

1. **Создать карту регионов** — новый файл `regions_map_<client_id>.json` на сервере (или загрузить через API).
2. **Зарегистрировать профиль** — POST `/clients` с телом профиля (или вручную добавить запись в таблицу `clients` в SQLite).
3. **Создать Google Таблицу** — скопировать шаблон таблицы заказчика, вставить `sheet_id` в профиль.
4. **Установить Apps Script** — скопировать `apps_script.gs` в новую таблицу, задать `WEBHOOK_URL` и секрет.
5. **Проверить** — запустить тестовый прогон через меню.

Пересборка Docker не нужна: сервер читает профили из БД динамически.

## 4. Sheets-интерфейс

### 4.1 Структура листов таблицы

```
Google Таблица клиента
├── Отчёт          — матрица-pivot (основной рабочий лист заказчика)
├── Данные         — плоский лог всех прогонов (для машины/фильтрации)
├── Настройки      — параметры прогона (читаются Apps Script)
└── Лог            — журнал запусков и статусов (опционально)
```

### 4.2 Лист «Настройки»

Формат: таблица ключ–значение (колонка A = ключ, колонка B = значение).

| Ключ | Тип | Описание | Пример |
|---|---|---|---|
| `client_id` | string | Выпадающий список клиентов | `sudheimer` |
| `depth` | enum | Глубина сбора | `10` |
| `with_labels` | bool | Включить разметку | `true` |
| `label_mode` | enum | Режим разметки | `snippets` |
| `date` | string | Дата сбора (`today` или ISO) | `today` |
| `force_relabel` | bool | Принудительная переразметка | `false` |
| `force_rebuild_report` | bool | Принудительная перестройка отчёта | `false` |
| `report_date` | string | Дата отчёта (`latest` или ISO) | `latest` |
| `provider_chain` | list[string] | Цепочка провайдеров (фолбек) | `zen` |

Колонка C — подсказка/описание параметра (необязательно, для удобства заказчика).

**Data Validation:** Для `depth` — список `10, 20, 50, 100`. Для `with_labels`, `force_relabel`, `force_rebuild_report` — список `true, false`. Для `label_mode` — список `domains, snippets, full`. Для `date` и `report_date` — свободный ввод с подсказкой формата.

### 4.3 Меню «SERPlux»

```
SERPlux
├── ▶ Запустить сбор
├── ⟳ Проверить статус
├── 📋 Построить отчёт за дату...
├── ─────────────────────
├── 👤 Клиенты
│   ├── Показать список клиентов
│   └── Добавить клиента...
├── ─────────────────────
└── ⚙ Настройки
    ├── Установить секрет
    ├── Показать текущий профиль
    └── 🔧 Управление провайдерами… (только админ)
```

### 4.4 Диалоги и потоки

#### «▶ Запустить сбор»

1. Apps Script читает лист «Настройки» → собирает параметры прогона.
2. Если `client_id` не задан → диалог с ошибкой «Выберите клиента в листе Настройки».
3. Если секрет не задан → диалог «Секрет не задан. Запустите SERPlux → Настройки → Установить секрет».
4. Показывает диалог подтверждения: «Запустить сбор для [client_name]? Глубина: [depth], Разметка: [with_labels], Дата: [date]» — кнопки «Запустить» / «Отмена».
5. POST `/run` → получает 202 Accepted.
6. Диалог: «Прогон запущен. Начало: [started_at]. Проверьте статус через меню SERPlux → Проверить статус».
7. Опционально: автоматически проверяет статус через 30 секунд (setTimeout в Apps Script).

#### «⟳ Проверить статус»

1. GET `/status` → получает `{status, started_at, message, finished_at}`.
2. Диалог со статусом:
   - `idle` → «Прогонов не было»
   - `starting` / `running` → «Прогон выполняется... Начало: [started_at]»
   - `ok` → «Прогон завершён успешно. Начало: [started_at], Конец: [finished_at]»
   - `error` → «Ошибка прогона: [message]»

#### «📋 Построить отчёт за дату...»

1. Диалог с полем ввода: «Введите дату (YYYY-MM-DD) или оставьте пустым для последней».
2. POST `/run` с `{client_id, report_only: true, report_date: <дата>}` — только построение отчёта без сбора.
3. Диалог с результатом.

#### «👤 Клиенты → Добавить клиента...»

1. Многошаговый диалог (серия `ui.prompt`):
   - Шаг 1: «Введите ID клиента (латиница, без пробелов)»
   - Шаг 2: «Введите отображаемое имя клиента»
   - Шаг 3: «Введите project_id в Topvisor»
   - Шаг 4: «Введите Google Sheet ID таблицы клиента»
2. POST `/clients` с собранными данными.
3. Диалог: «Клиент [name] добавлен. Не забудьте загрузить карту регионов на сервер».

#### «⚙ Настройки → Показать текущий профиль»

1. GET `/clients/<client_id>` → получает профиль.
2. Диалог с текстовым представлением профиля (ключ: значение).

#### «⚙ Настройки → 🔧 Управление провайдерами…»

1. GET `/providers` → получает список провайдеров.
2. Диалог с таблицей: имя провайдера | модель | статус (вкл/выкл) | приоритет | стоимость/1K.
3. Кнопки:
   - «Добавить…» → многошаговый диалог:
     - Шаг 1: «ID провайдера (латиница, напр. openai)»
     - Шаг 2: «Отображаемое имя»
     - Шаг 3: «Endpoint URL (OpenAI-совместимый)»
     - Шаг 4: «Название модели»
     - Шаг 5: «Имя переменной в .env для API-ключа»
     - Шаг 6: «Стоимость за 1K токенов (USD, необязательно)»
   - «Вкл/Выкл» → переключает enabled
   - «↑ / ↓» → меняет priority
   - «Удалить» → soft delete (только если не используется в активной цепочке)
4. POST `/providers` для добавления, PUT `/providers/{id}` для изменений.

**Примечание:** API-ключи НЕ передаются через UI — они задаются в `.env` на сервере. В профиле провайдера хранится только имя переменной (`api_key_env_var`).

### 4.5 Индикация статуса

Поскольку Apps Script не поддерживает фоновые уведомления, статус отображается:

- **Ячейка-статус** на листе «Настройки» (строка `status`, колонка B): Apps Script обновляет её после запуска и после проверки статуса. Значения: `idle`, `running`, `ok`, `error`. Цветовая заливка: серый / жёлтый / зелёный / красный.
- **Лист «Лог»** (опционально): Apps Script дописывает строку с датой, клиентом, статусом и сообщением после каждого запуска/проверки.
- **Лист «Лог»** — колонка `provider_used`: какой провайдер фактически использовался для разметки (первый в цепочке или фолбек).

### 4.6 Ограничения Apps Script

- Нет push-уведомлений: заказчик должен вручную нажать «Проверить статус» или подождать.
- Таймаут Apps Script — 6 минут. Прогон идёт на сервере в фоне, Apps Script только запускает и проверяет.
- Диалоги (`ui.alert`, `ui.prompt`) блокирующие — нельзя показать прогресс-бар.

## 5. API-контракт

### 5.1 Текущее состояние

Сейчас реализованы три эндпоинта:

- `GET /health` — health-check, без авторизации
- `GET /status` — статус последнего прогона, Bearer-токен
- `POST /run` — запуск пайплайна, Bearer-токен; тело: `{regions_map, with_labels, depth}`

### 5.2 Целевой контракт `/run`

**Метод:** `POST /run`
**Авторизация:** `Authorization: Bearer <WEBHOOK_SECRET>`
**Content-Type:** `application/json`

#### Тело запроса (целевое)

```json
{
  "client_id": "sudheimer",        // НОВОЕ: ID клиента (обязателен при мультиклиентности)
  "depth": 10,                     // ЕСТЬ: глубина сбора (10/20/50/100)
  "date": "today",                 // НОВОЕ: дата сбора ("today" или "YYYY-MM-DD")
  "with_labels": true,             // ЕСТЬ: включить разметку
  "label_mode": "snippets",        // НОВОЕ: режим разметки ("domains"/"snippets"/"full")
  "force_relabel": false,          // НОВОЕ: игнорировать кэш меток
  "force_rebuild_report": false,   // НОВОЕ: перестроить отчёт даже если уже есть
  "report_date": "latest",         // НОВОЕ: дата для построения отчёта
  "report_only": false,            // НОВОЕ: только построить отчёт, без сбора
  "provider_chain": ["zen"],         // НОВОЕ: фолбек-цепочка провайдеров (дефолт из профиля)
  "regions_map": "regions_map.json" // УСТАРЕВАЕТ: заменяется на client_id → профиль
}
```

**Обратная совместимость:** Поле `regions_map` сохраняется для совместимости с текущим Apps Script. При наличии `client_id` — `regions_map` игнорируется (берётся из профиля).

#### Ответ 202 Accepted

```json
{
  "accepted": true,
  "run_id": "2026-07-02T10:00:00Z",
  "started_at": "2026-07-02T10:00:00.123456+00:00",
  "client_id": "sudheimer"
}
```

#### Ответ 409 Conflict (прогон уже идёт)

```json
{
  "detail": "Прогон уже выполняется, подождите завершения"
}
```

## 6. Открытые вопросы к заказчику

Решения, которые нельзя принять без Михаила.

### 6.1 Мультиклиентность

**Q1.** Сколько клиентов планируется в ближайшие 6 месяцев? Это определяет, нужна ли единая таблица с выпадающим списком или достаточно отдельной таблицы на каждого клиента.

**Q2.** Должны ли все клиенты быть в одной Google Таблице (одно место управления) или в разных таблицах (изоляция, проще настроить)?

**Q3.** Кто будет добавлять новых клиентов — только Михаил или другие пользователи? Это влияет на сложность интерфейса добавления.

---

### 6.2 Режимы разметки

**Q4.** Нужны ли все три режима разметки (`domains`, `snippets`, `full`) с переключателем в интерфейсе, или достаточно одного режима по умолчанию (`snippets`)? Переключатель усложняет интерфейс.

**Q5.** Если режим `domains` нужен: кто ведёт списки доменов (позитивные/негативные/нейтральные) для каждого клиента? Где они хранятся — в Google Sheets, в файле на сервере, в БД?

**Q6.** Если режим `full` (заход на страницу) нужен: приемлем ли бюджет? Стоимость значительно выше `snippets` (нужен HTTP-запрос к каждому URL + больший контекст для LLM). Также: как обрабатывать страницы с блокировкой ботов?

---

### 6.3 Параметр `date` (ретроспективный сбор)

**Q7.** Нужен ли сбор выдачи за прошлые даты (не только «сегодня»)? Если да — нужно проверить, поддерживает ли Topvisor Snapshots API получение снимка за произвольную прошлую дату. Если не поддерживает — параметр `date` будет работать только для фильтрации уже собранных данных при построении отчёта.

---

### 6.4 Отчёт и версионность

**Q8.** Нужен ли «Лог» как отдельный лист в таблице (история запусков: дата, клиент, статус, кол-во строк)? Или достаточно статуса в диалоге?

**Q9.** Нужна ли возможность построить отчёт за конкретную прошлую дату из уже собранных данных (без нового сбора)? Данные в БД есть, нужен только UI-триггер.

**Q10.** Текущий формат отчёта (матрица 16 колонок, 4 субъекта) захардкожен под одного клиента. При добавлении второго клиента с другой структурой — нужна ли полная параметризация раскладки колонок, или второй клиент может иметь другой формат отчёта?

---

### 6.5 Безопасность и доступ

**Q11.** Нужен ли разграниченный доступ — разные секреты для разных клиентов, или один общий `WEBHOOK_SECRET` для всех?

**Q12.** Кто имеет доступ к листу «Настройки» в таблице? Если таблица расшарена с клиентом — он может случайно изменить параметры.

---

### 6.6 Деплой и обновления

**Q13.** Как сейчас происходит обновление кода на сервере? Нужен ли механизм обновления без остановки сервиса (zero-downtime deploy)?

**Q14.** Нужен ли мониторинг/алерты при падении сервиса (например, Telegram-уведомление при `status=error`)?

---

### 6.7 LLM-провайдеры

**Q18.** Как определять «качество» разметки провайдера для сравнения? Варианты:
- Ручная валидация выборки (админ проверяет N случайных разметок).
- Консенсус между провайдерами (если 2 из 3 согласны — засчитывается).
- Эталонный набор данных с заранее размеченными ответами.

**Q19.** Нужен ли лимит бюджета на провайдера (макс. стоимость за прогон)? Если да — какой дефолт и как уведомлять о превышении?

---
