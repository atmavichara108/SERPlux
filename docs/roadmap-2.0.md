## Кандидаты в v2.0

### 1. Deep-режим разметки (реализация)

**Зачем:** Уточнение neutral результатов по полному контенту страницы, повышение качества разметки.

**Текущее состояние:** Заглушка — `labeler.py::_label_group_deep()` возвращает строки без изменений.

**Связь с v1.0:** Лист «Спорные» в шаблоне таблицы — задел для этого режима. Deep-режим будет:
1. Читать лист «Спорные» (накопитель сомнительных случаев)
2. Заходить на страницу по URL
3. Парсить контент
4. Переразмечивать neutral → positive/negative

**DoD:**
- HTTP-запрос к каждому URL с neutral sentiment
- LLM по полному тексту страницы (не сниппету)
- Сохранение с `source='page'` в domain_labels
- Синхронизация с листом «Спорные» (чтение → обновление)
- Тесты: покрытие happy path + ошибки HTTP + таймауты
- Документация: обновить `docs/labeling_canon.md`, `docs/contracts.md`, `docs/user-guide.md`

### 1.5. Изоляция эталона по клиенту (мультиклиентность)

**Зачем:** Сейчас разметка (domain_labels) общая для всех клиентов. Два клиента с одним доменом делят одну метку. При изоляции — каждый клиент видит свой кэш, нет пересечений.

**Текущее состояние:** 
- БД `domain_labels` с ключом `(domain, query, geo)` без `client_id`
- Один клиент может «заразить» кэш другого

**DoD:**
- Миграция: добавить `client_id` в PRIMARY KEY `domain_labels` → `(client_id, domain, query, geo)`
- SQL миграция: добавить колонку `client_id` + заполнить существующие записи (маппинг по профилям)
- Правки кода:
  - `storage.py::get_cached_label()` → фильтр по `client_id`
  - `storage.py::save_label()` → добавить `client_id` в INSERT
  - `labeler.py::_label_one_llm()` → передать `client_id`
  - `webhook.py::/labels/import` → добавить `client_id` в контракт
- Тесты: проверка что разметка клиента А не видна клиенту Б
- Документация: обновить `docs/contracts.md`, `docs/CANON.md` (мультиклиентность)
- **Гарантия:** обратная совместимость (старые записи без client_id остаются в archive-таблице)

### 2. Удаление legacy label_mode

**Зачем:** Убрать риск «зомби-меток» с устаревшими режимами `domains`/`snippets`/`full`.

**Текущее состояние:** `storage.LABEL_MODES` и CHECK `labels.label_mode` допускают legacy-режимы. Новый код работает только с `auto`/`deep`.

**DoD:**
- Миграция: перенести существующие метки `domains`/`snippets`/`full` в `auto`/`deep` (или пометить как архивные)
- Сузить `storage.LABEL_MODES` до `{"auto", "deep"}`
- Изменить CHECK `labels.label_mode` на `('auto','deep')`
- Обновить `migrate.py` и тесты
- Документация: обновить `docs/labeling_canon.md`

### 3. Автоматизация эталона

**Зачем:** Убрать ручной запуск `parseList1ToEtalon()` в редакторе Apps Script.

**Текущее состояние:** Функция запускается вручную через Run. Не добавлена в меню onOpen.

**DoD:**
- Вариант A: добавить пункт меню «Обновить эталон из Лист1» (требует подтверждения)
- Вариант B: cron-триггер (раз в неделю/месяц) с уведомлением в Telegram/Email
- Логирование: число обновлённых записей, ошибки
- Тесты: мок Лист1 → проверка «Эталон разметки»
- Документация: обновить `docs/CANON.md`, `docs/user-guide.md`

---

### 4. Ротация бэкапов по cron

**Зачем:** Автоматическая очистка старых бэкапов (старше 30 дней).

**Текущее состояние:** `backup_db.sh` ротирует только последние 10 при ручном вызове.

**DoD:**
- Cron-задача на сервере: `0 3 * * * /root/serp/backup_db.sh --rotate 30`
- Уведомления: Telegram/Email при ошибке бэкапа
- Логирование: размер бэкапа, время выполнения, число удалённых файлов
- Документация: обновить `docs/verification.md`, `docs/deploy.md`

---

### 5. Watchdog для зависших прогонов

**Зачем:** Автоматическая пометка зависших прогонов как `stalled`/`error`.

**Текущее состояние:** Статус персистентен (`run_status`), но не устаревает.

**DoD:**
- Фоновый поток или cron (раз в 15 минут)
- Проверка: `status='running'` и `updated_at` старше порога (2 часа)
- Перевод в `stalled`/`error` с сообщением "прогон завис"
- API/меню: ручной сброс статуса
- Тесты: мок `run_status` → проверка перехода в `stalled`
- Документация: обновить `docs/contracts.md` (новый статус)

---

### 6. Уведомления о завершении прогона

**Зачем:** Оператор получает уведомление в Telegram/Email при завершении прогона.

**Текущее состояние:** Нет. Оператор проверяет статус вручную через меню или `/status`.

**DoD:**
- Интеграция: Telegram Bot API или SMTP
- События: прогон завершён успешно / прогон упал с ошибкой
- Настройки: webhook URL / chat_id / email в листе «Настройки»
- Тесты: мок HTTP-клиента → проверка отправки
- Документация: обновить `docs/user-guide.md`

---

### 7. Мониторинг стоимости LLM

**Зачем:** Контроль расходов на DeepSeek/Zen.

**Текущее состояние:** Нет учёта токенов и стоимости.

**DoD:**
- Таблица `llm_stats`: date, client_id, model, tokens_in, tokens_out, cost_usd
- Запись после каждого вызова LLM в `labeler.py`
- API: `GET /stats/llm?date=...&client_id=...`
- UI: лист «Статистика» в Google Sheets (опционально)
- Документация: обновить `docs/contracts.md`
