# Тестирование инфраструктуры и shell-скриптов

Документ описывает, какие части инфраструктуры можно проверять локально и в CI,
а какие требуют реального Docker-окружения и ручного тестирования на сервере.

## Что у нас есть сейчас

### 1. Python unit/integration тесты

- **Инструмент:** `pytest`
- **Запуск локально:** `python -m pytest -q`
- **Запуск в CI:** `.github/workflows/ci.yml` job `test`
- **Что покрывают:** логика модулей (`storage.py`, `webhook.py`, `collector.py`,
  `labeler.py`, `main.py`, `migrate.py` и др.) без внешних API
- **Что НЕ покрывают:** Docker, shell-скрипты, реальные Topvisor/Google Sheets API

### 2. Shellcheck — статический анализ shell-скриптов

- **Инструмент:** `shellcheck`
- **Запуск в CI:** `.github/workflows/ci.yml` job `shellcheck`
- **Что проверяет:** синтаксис, неинициализированные переменные, `set -e` +
  арифметика, `grep` без совпадений, переносимость, кавычки
- **Что НЕ проверяет:** логику взаимодействия с Docker, реальное поведение
  контейнеров, порядок шагов

### 3. Синтаксическая проверка bash

- **Инструмент:** `bash -n`
- **Запуск локально:** `bash -n verify.sh backup_db.sh deploy.sh`
- **Что проверяет:** базовый синтаксис
- **Что НЕ проверяет:** семантику, `set -e` ловушки, отсутствие зависимостей

## Что нельзя проверить локально

### Docker-интеграция

Агент (build/plan) работает в окружении без доступа к Docker API:

```bash
permission denied while trying to connect to the docker API
at unix:///var/run/docker.sock
```

Это значит, что агент **не может**:
- собрать Docker-образ
- запустить контейнер
- выполнить `docker compose exec`
- проверить `verify.sh` в реальном контейнере
- проверить `backup_db.sh` с ротацией
- проверить `deploy.sh` end-to-end

### Почему так

Docker daemon работает на хосте, а агент — в изолированном процессе без
доступа к `/var/run/docker.sock` и без прав в группе `docker`. Дать агенту
такой доступ рискованно: он может случайно или злонамеренно изменить
производственные контейнеры.

## Решение: что нужно для полного локального тестирования

### Вариант A: Docker-in-Docker (DinD) для агента

Запускать агента внутри контейнера с собственным Docker daemon (DinD) или с
доступом к sibling Docker socket.

**Плюсы:**
- Агент может собирать образы и тестировать `verify.sh` локально
- Можно добавить интеграционные тесты для shell-скриптов

**Минусы:**
- Требует привилегированного контейнера (`--privileged`)
- Увеличивает сложность инфраструктуры
- Риски безопасности (privilege escalation)

**Применимость:** подходит для выделенного CI-агента, но не для повседневного
development-агента.

### Вариант B: Docker API через удалённый хост

Настроить для агента отдельный Docker host (например, `tcp://docker-test:2375`
или SSH), где разрешены только тестовые контейнеры.

**Плюсы:**
- Агент может тестировать без привилегий на локальной машине
- Можно изолировать тестовые контейнеры от production

**Минусы:**
- Требует отдельного тестового Docker-хоста
- Сетевые задержки и сложность настройки
- Не решает проблему отсутствия Docker на машине агента

### Вариант C: Моки Docker CLI в тестах

Для shell-скриптов писать unit-тесты с подменой `docker` на mock-скрипт.

**Плюсы:**
- Можно проверить логику `verify.sh` без реального Docker
- Быстро и безопасно

**Минусы:**
- Не ловит проблемы, связанные с реальным Docker Compose (формат JSON,
  версии, права доступа)
- Требует поддержки тестовых фикстур

**Применимость:** хорошо для быстрой регрессии, но не заменяет
интеграционное тестирование на сервере.

### Вариант D: Shell-интеграционные тесты на сервере (выбранный подход)

Агент пишет код и shell-скрипты, а человек выполняет их на сервере и
присылает вывод. Агент анализирует результат и делает правки.

**Плюсы:**
- Не требует изменения окружения агента
- Тестирование происходит на реальном сервере с реальным Docker
- Минимальные риски безопасности

**Минусы:**
- Медленнее, чем локальное тестирование
- Требует участия человека
- Не автоматизирует проверку до push

**Применимость:** текущий подход проекта. Для автоматизации — добавить
shellcheck в CI (сделано) и рассмотреть Docker-based CI runner (Вариант A или
B) в будущем.

## Текущая стратегия

1. **Локально / CI:**
   - `pytest` — Python модули
   - `shellcheck` — shell-скрипты
   - `bash -n` — синтаксис shell

2. **На сервере (человек):**
   - `./deploy.sh` — end-to-end деплой
   - `./verify.sh` — интеграционная верификация
   - `./backup_db.sh` — бэкап и ротация
   - Ручные проверки функциональности (Google Sheets, сбор)

3. **GitHub Actions:**
   - Блокирует merge, если `pytest` или `shellcheck` падают
   - Не имеет доступа к серверу и не делает деплой

## Чек-лист для агента при изменении shell-скриптов

Перед push изменения в `.sh` файлах:

- [ ] `bash -n verify.sh backup_db.sh deploy.sh` — OK
- [ ] Проверить все `grep` под `set -e`: есть ли `|| true`
- [ ] Проверить все `((var++))` под `set -e`: заменены на `$((var + 1))`
- [ ] Проверить все подстановки `$(cmd | grep)` под `set -e`
- [ ] Проверить использование `jq` — если да, нужен fallback без jq
- [ ] Проверить, что команды не зависят от локальных инструментов сервера
- [ ] Обновить `docs/verification.md` и `docs/infra-testing.md`
- [ ] После push попросить пользователя запустить `./verify.sh` на сервере

## Известные ловушки `set -e` в bash

### 1. Арифметика: `((var++))`

```bash
set -e
count=0
((count++))   # exit code 1, скрипт умирает
```

**Решение:**
```bash
count=$((count + 1))
```

### 2. `grep` без совпадений

```bash
set -e
count=$(docker logs ... | grep -i error | wc -l)   # exit code 1
```

**Решение:**
```bash
count=$(docker logs ... | { grep -i error || true; } | wc -l)
```

### 3. Подстановка команд с ненулевым exit code

```bash
set -e
output=$(docker compose exec ... python3 -c "..." 2>&1)
# Если python3 вернул 1, скрипт умирает сразу
```

**Решение:**
```bash
set +e
output=$(docker compose exec ... python3 -c "..." 2>&1)
exit_code=$?
set -e
```

### 4. `echo` с переменными

```bash
set -e
msg="-n something"
echo "$msg"   # интерпретирует -n как флаг
```

**Решение:**
```bash
printf '%s\n' "$msg"
```

## Рекомендации по развитию

### Ближайшие (низкая стоимость)

1. **Shellcheck в CI** — сделано (`.github/workflows/ci.yml`)
2. **Shellcheck в pre-commit** — добавить `.pre-commit-config.yaml` для
   локальной проверки перед коммитом
3. **Мок-тесты для verify.sh** — создать `tests/test_verify.sh` с mock
   `docker` и `curl`, чтобы проверить логику без Docker

### Средние (требуют инфраструктуры)

4. **Docker-based CI runner** — GitHub Actions self-hosted runner на
   тестовом сервере, где можно запускать `docker compose` и `verify.sh`
5. **Test-container job** — в CI поднимать минимальный контейнер и
   прогонять `verify.sh` против него

### Долгосрочные

6. **Полная staging-среда** — отдельный сервер/контейнер, идентичный
   production, для автоматического интеграционного тестирования
