# CANON.md — источник истины SERPlux

> Этот документ обновляется в конце каждой сессии, где меняется геометрия, контракт или список граблей.
> Он — первое, что читается в начале новой сессии.

## Мультиклиентность (v1.0+)

### Модель

**Заказчик** управляет несколькими **проектами в Topvisor** (каждый проект = отдельный клиент SERPlux).

- **Один заказчик** → несколько **клиентов** (профили в БД)
- **Один клиент** → одна **Google-таблица** (привязка: sheet_id в профиле клиента + project_id Topvisor)
- **Мастер-таблица** (текущая рабочая) → **источник шаблона** для новых клиентов
- **Единая структура** для шаблона и всех клиентов: **6 листов без Лист1**

### Структура листов клиента

Каждая таблица клиента содержит:

| Лист | Назначение | Динамика |
|------|-----------|----------|
| **Настройки** | Параметры сбора (client_id, depth, режимы) | Ручное редактирование |
| **Данные** | Кэш выдачи (плоский формат) | Перезапись при каждом прогоне |
| **Отчёт** | Матрица позиций по гео (накопительный режим) | Новые версии сверху, max 10 |
| **Эталон разметки** | Кэш разметки по URL (url, query, geo) | Пополняется нейронкой при прогонах |
| **Спорные** | Накопитель для deep-режима (v2.0) | Читается в future версии для доразметки |
| **Лог** | История событий | Логируются запуски, ошибки |

**Лист1 исключён** из структуры (каноническая раскладка отчёта сохранена в этом документе и на листе «Отчёт»).

### Размер матрицы (динамический)

- **Число гео:** столько, сколько выбрано в профиле клиента (regions_map) — НЕ ограничено 6
- **Число субъектов (запросов):** столько, сколько добавлено в профиль клиента (queries) — НЕ ограничено
- **Глубина (depth):** 10, 20, 50 или 100 — настраивается в листе «Настройки» для каждого прогона
- **Раскладка:** `_build_subject_layout()` в reporter.py динамически расширяет матрицу, не трогать

### Кэш разметки (БД)

**Текущее состояние (v1.0):**
- Кэш `domain_labels` общий для всех клиентов (ключ: `(url, query, geo)` — полный URL, не домен)
- Два клиента с одним URL делят одну метку
- Матчинг по точному совпадению URL (нормализация не требуется)

**Планируется (v2.0):**
- Изоляция по клиенту: ключ → `(client_id, url, query, geo)`
- Каждый клиент видит только свой кэш (см. roadmap-2.0.md пункт 1.5)

### Инициализация шаблона

Функция **`initTemplateSheets()`** в apps_script.gs:
- Создаёт 6 листов (если их нет) с заголовками
- Идемпотентна: существующие листы не дублирует
- Запускается вручную один раз: Apps Script → Run → `initTemplateSheets()`

Подробнее: docs/onboarding-client.md, template/SHEETS.md

## Геометрия Лист1 (матрица разметки)

Лист1 — матричная структура, источник истины для эталона разметки.

### Вертикальная структура

| Строка | Содержимое |
|--------|-----------|
| 1 | Заголовок версии: `Позиции {провайдер} на {ДД.ММ.ГГГГ}` |
| 2 | Пустой буфер |
| 3 | Имя субъекта в **правой** колонке блока (C, H, K, …) |
| 4 | Гео-подзаголовок (Lithuania / Germany / United Kingdom / …) в **левой** колонке блока |
| 5–14 | Номера позиций 1..depth (depth=10) в левой колонке, URL/домены в правой |
| 15 | Пустой буфер |
| 16+ | Повтор для следующего гео: строка гео → depth номеров → буфер |

После последнего гео — два пустых строки-разделителя перед следующим провайдером.

### Горизонтальная карта колонок (0-indexed)

| Субъект | Левая колонка (pos) | Правая колонка (url) | Буфер перед | Буфер после |
|---------|---------------------|----------------------|-------------|-------------|
| 1 | 1 (B) | 2 (C) | — | 3 колонки (3,4,5 = D,E,F) |
| 2 | 6 (G) | 7 (H) | 1 колонка (5→6) | 1 колонка (8 = I) |
| 3 | 9 (J) | 10 (K) | 1 колонка (8 = I) | 1 колонка (11 = L) |
| 4 | 12 (M) | 13 (N) | 1 колонка (11 = L) | 1 колонка (14 = O) |
| N (≥2) | 6 + (N-2)*3 | 7 + (N-2)*3 | 1 колонка | 1 колонка |

Формула для субъекта N (N ≥ 2):
- `pos = 6 + (N - 2) * 3`
- `url = pos + 1`

Для N=1: `pos = 1, url = 2`.

### Содержимое колонок

- **Левая колонка (pos):** гео-подзаголовок (строка 4) + номера позиций 1..depth (строки 5–14) + заливка sentiment на ячейке номера.
- **Правая колонка (url):** имя субъекта (строка 3, крупный шрифт) + URL/домены напротив каждого номера (строки 5–14).
- **Буферные колонки:** пустые, без данных и заливки.

### Заливка sentiment

Заливка применяется **к ячейке номера позиции** (левая колонка), НЕ к URL и НЕ ко всей строке.

| Цвет | Sentiment |
|------|-----------|
| Зелёный | positive |
| Красный | negative |
| Жёлтый | neutral |

## Корневая причина проблем эталона

Лист «Эталон разметки» был сгенерён **сломанным парсером Лист1**:
- В `query` попали страны (united arab emirates, cyprus eng…) вместо имён субъектов.
- В `geo` записана константа `United Kingdom` вместо реальных стран.
- Имена субъектов (juri sudheimer, erik sudheimer, sct chemicals…) потеряны.
- Из-за этого ~420 размеченных ячеек схлопнулись в ~81 битую запись.
- Ключ `(domain, query, geo)` не совпадает с реальными прогонами → кэш не работает.

## Правило процесса

Этот CANON.md обновляется в конце каждой сессии, где меняется геометрия, контракт или список граблей. Он — первое, что читается в начале новой сессии.

### Проверка тестов: обязательно в чистом окружении

Все тесты должны проходить **независимо от реальных переменных окружения** (GOOGLE_SHEET_ID, GOOGLE_CREDENTIALS_PATH и т.п.). 

**Правило:** Если тесты проходят локально, но падают в CI — это скрытая зависимость от среды.

**Решение:** патчируем функции напрямую, не полагаясь на env-переменные.

**Пример:** в `test_reporter.py` раньше тесты падали в CI, потому что `reporter._get_spreadsheet()` проверяет наличие `GOOGLE_SHEET_ID` ДО вызова gspread. Решение: патчим `reporter._get_spreadsheet` полностью, обходя все проверки окружения.

**Проверка тестов (в чистом окружении):**
```bash
unset GOOGLE_SHEET_ID && unset GOOGLE_CREDENTIALS_PATH && python -m pytest tests/ -q
```
Должно быть зелёно (224 passed).

## Поведение reporter (накопительный режим)

Отчёт накапливается сверху: каждый новый прогон вставляет блок СВЕРХУ листа «Отчёт», старые версии сдвигаются вниз.

### Признак начала версии

Строка, где первая ячейка начинается с `"Позиции "` и содержит `" на "`.
Пример: `"Позиции Google на 11.7.2026"`.

### Хранение версий

- Максимум: 10 версий (константа `MAX_VERSIONS` в `reporter.py`).
- После вставки нового блока сверху считается число версий по заголовкам.
- Если > 10 — самые старые версии (снизу) удаляются через `deleteDimension`.

### Механика вставки

1. Собрать `report_data` нового прогона (логика матрицы/гео/субъектов не меняется).
2. Прочитать текущее содержимое листа (`worksheet.get_all_values()`).
3. Если есть данные — вставить строки сверху через `insertDimension` (один batch_update запрос).
4. Записать новый блок в `A1` (`worksheet.update`).
5. Применить заливку к новому блоку (строки 0..N-1).
6. Обрезать старые версии, если их > 10.

### Сдвиг форматирования

`insertDimension` в Google Sheets сдвигает форматирование (включая заливку) автоматически вместе с данными. Заливку нового блока накладываем ПОСЛЕ вставки, по актуальным (верхним) координатам.
